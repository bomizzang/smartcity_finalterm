# 판교 vs 김포한강 자족용지 비교분석 시스템

가천대학교 스마트시티학과 기말과제 — "데이터로 진단하는 업무지구의 성공과 실패"

## 폴더 구조

```
site/
├── index.html        ← 메인 페이지
├── js/app.jsx         ← 화면 코드 (React, 브라우저에서 직접 변환되어 동작)
└── data/              ← 모든 통계·지도 데이터 (json, geojson)
```

## 사용법 (npm, 빌드 불필요)

1. GitHub에 새 저장소(Public) 만들기
2. 이 폴더(`index.html`, `js/`, `data/`) 안의 내용 전체를 저장소 루트에 업로드
3. 저장소 **Settings → Pages** → Source를 "Deploy from a branch" → `main` 브랜치, `/ (root)` 폴더로 설정
4. 몇 분 후 `https://본인계정.github.io/저장소이름/` 에서 확인

로컬 컴퓨터에서 미리 보고 싶다면, `index.html`을 그냥 더블클릭해서 열어도 대부분 동작하지만,
일부 브라우저는 로컬 파일에서 `fetch()`로 `data/` 폴더를 읽는 것을 막을 수 있다(보안 정책).
이 경우 VS Code의 "Live Server" 확장 또는 `python3 -m http.server` 명령으로 로컬 서버를 띄워서 확인하면 된다.

## 지도 타일(VWorld) 안내

`js/app.jsx` 안에 VWorld 인증키가 이미 들어가 있어 바로 동작한다.
- 키가 막히거나 만료되면 자동으로 OSM 무료 타일로 대체된다.
- 이 키를 다른 용도로 재사용하지 않는 것을 권장. 필요하면 본인 명의로 새로 발급(vworld.kr)해서 교체.

## 구역계 정의 근거

처음엔 청라국제업무지구(인천)로 분석을 시작했는데, 건축HUB에서 받은 건축물대장을 확인해보니
지정된 9개 블록(27.8ha) 전체에 건축물 자체가 거의 없어서("물리적 조성 완료" 조건을 충족하지
못해서) 비교 대상을 김포한강신도시 자족시설용지로 바꿨다.

- **판교테크노밸리**: 제1판교테크노밸리만을 분석 대상으로 한다. 제2판교테크노밸리(성남시 수정구 시흥동·금토동 일원, 약 43만㎡)는 행정구역상 분당구가 아니며 우리 경계(분당구 삼평동, 위도 37.40 부근) 바깥에 위치해 제외했다. OSM 경계 기반으로 추출 후 금토천 하천부지(버퍼 66.5m)를 제외하여 보정했다. 보정한 면적(66.0ha)이 제1판교 공식 단지 면적(66만㎡)과 정확히 일치해서 경계가 맞다는 걸 확인했다.
- **김포한강신도시 자족시설용지**: 구래동 6871번지 일대(17.35ha)로 정의했다. 건축HUB 건축물대장을 조회해보니 이 번지 내 모든 건물이 업무시설·공장(지식산업센터)·근린생활시설로만 구성되어 있어 자족시설용지 블록으로 보고 확정했다. 실제로 이 단지의 분양 광고에서도 "김포한강신도시 자족시설MS-13-1블럭"이라는 공식 블록명이 확인됐다.

## 분석 방법론 메모

- **토지이용**: 연속지적도(SHP) 필지와 분석 경계의 면적 교차비율이 50% 이상인 필지만 채택했다(다수겹침 원칙). 건축물대장과 번지수로 매칭했다. 다만 건축HUB의 "건축물대장 표제부" 데이터는 집합건물(여러 동·호로 구성된 단지)의 경우 동/호별 레코드에는 대지면적·건폐율·용적률이 0으로 기록되어 있어("총괄표제부"에만 단지 전체 값이 들어있는 구조), 판교 9건·김포 1건에서 "건물은 있는데 건폐율·용적률이 0%"인 이상치가 발견됐다. 이 부분은 건축HUB에서 "총괄표제부" 데이터를 별도로 받아 보정했다. 단, 분석에 쓰인 핵심 지표(평균 용적률 663.3%/658.2% 등)는 연면적 합계 ÷ 대지면적 합계로 계산했기 때문에 이 이상치의 영향을 받지 않았다 — 영향은 지도 클릭 시 보이는 개별 건물 속성에 한정됐다.
- **인구·종사자**: SGIS 집계구 경계와 분석 경계의 면적 비례 배분 방식으로 산출했다. 우리 경계가 SGIS 집계구보다 훨씬 작아서, 한 집계구가 경계 안에 몇 % 걸치는지를 계산해 그 비율만큼만 인구·종사자 수치를 가져왔다.
  - 공식: `구역 내 추정값 = Σ (집계구 통계값 × (구역 ∩ 집계구 면적) / 집계구 전체 면적)`
- **등시간권**: subway_network.zip의 노드/링크 그래프를 다익스트라 알고리즘으로 최단시간 탐색했다. 도달역에 1km 버퍼를 씌우고 시군구 경계와 면적비례로 결합해 인구·종사자를 추정했다.
- **산업분류**: SGIS 10차 표준산업분류 대분류(19개) 코드가 KSIC 표준 순서(A~S)와 같다는 점을 이용해 업종명을 매핑했다.
- **도로망 밀도**: OSM 차도 분류(motorway~unclassified)만 경계로 클립해서 길이를 합산한 뒤 km/km²로 환산했다.
- **역세권 비율 / 버스정류장 밀도**: 핵심역 1km 버퍼와 경계의 교차비율, 그리고 국토교통부 버스정류장 위치정보 중 경계 안에 포함되는 정류장 수로 계산했다.

### 토지이용 혼합도(LUM) 산출 방식 — 두 가지 기준과 그 차이

LUM(Land Use Mix)은 분석 단위·용도 분류 기준에 따라 값이 크게 달라지는 지표라, 본 시스템은 **건축물대장 주용도 기준**을 메인 지표로 채택하되 **용도지역 기준**도 함께 산출해 비교한다.

**1) 건축물대장 주용도 기준 (메인 지표, comparison_stats.json의 `lum_entropy`)**

건축물대장 주용도 4종(업무시설/교육연구시설/자동차관련시설/공동주택)의 연면적 비중을 Shannon Entropy로 정규화한다.

```
LUM = -Σ(pᵢ × ln(pᵢ)) / ln(n)
```
- pᵢ = 주용도 i의 연면적 비중, n = 0이 아닌 주용도 카테고리 수
- 0~1로 정규화되어 지역 간 직접 비교가 쉽다 (0 = 단일 용도, 1 = 완전 균등 혼합)
- 결과: **판교 0.574 / 김포 0.769**

**2) 용도지역(zoning) 기준 (참고 지표, 정규화 없음)**

VWorld 토지이용계획도의 용도지역 구성비에서 도로·공원·주차장 등 비건축 용도를 제외한 뒤, **정규화하지 않은(raw) Shannon Entropy**를 그대로 사용하는 방식이다.

```
LUM_raw = -Σ(pᵢ × ln(pᵢ))   (※ ln(n)으로 나누지 않음)
```
- 제외 대상: 도로, 공원/녹지, 주차장, 광장 등 (구체적으로 어떤 카테고리를 "용도지역"으로 볼지에 따라 결과가 갈린다)
- 우리 데이터로 같은 방식을 적용하면: **판교 ≈ 0 (도시지원시설용지 59.3%가 사실상 단일 카테고리로 남아 entropy가 0에 수렴) / 김포 ≈ 0.37**

**두 기준이 이렇게 다른 이유:**
1. *측정 대상이 다르다* — 건축물대장 기준은 "실제 지어진 건물의 용도 다양성", 용도지역 기준은 "도시계획상 지정된 토지 용도 다양성"을 본다.
2. *정규화 여부가 다르다* — 건축물대장 기준은 0~1로 정규화해 카테고리 수에 무관하게 비교 가능하지만, 용도지역 raw entropy는 정규화하지 않아 카테고리 수와 분포 형태에 따라 절대값 스케일 자체가 다르다.
3. *제외 기준이 다르다* — 용도지역 raw entropy 방식은 도로·공원 등 "비건축 용도"를 빼고 재정규화하는데, 어디까지를 제외 대상으로 볼지가 분석자마다 달라질 수 있다.

**메인 지표로 건축물대장 기준을 택한 근거**: 과제 평가기준(§3-1)의 "토지이용 혼합도" 행은 산출 방법을 "위 데이터 가공"으로 명시하고 있고, 바로 위 행이자 같은 표 안에서 §3-0부터 세트로 지정된 데이터가 "토지이용계획(용도지역) + 건축물대장"이다. 본 시스템은 그중 정규화가 가능해 지역 간 비교가 직관적인 건축물대장 연면적 기준을 메인으로 채택했다. 동일 데이터를 다른 기준(용도지역, 비정규화)으로 산출하면 결과가 크게 달라질 수 있다는 점을 위와 같이 투명하게 공개한다.

실행 가능한 스크립트는 `scripts/` 폴더에 있다 (자세한 사용법은 `scripts/README.md` 참고).
